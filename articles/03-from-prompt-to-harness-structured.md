# 从 Prompt 到 Harness：AI Agent 的运行层是怎么长出来的


## 副标题:为什么真正值钱的不是会说话的模型,而是能让模型变成劳动者的系统

先抛一句总判断,接下来的所有内容都围绕它展开:

**AI Agent 的历史,不是模型越来越聪明的历史,而是人类不断给模型补上"工作现场"的历史。**

这个判断很重要。因为过去两年,AI 圈有一个普遍的误会:大家总以为 Agent 的进化路线是模型更大一点、推理更强一点、工具更多一点,Agent 就自然能干活。

这个想法很诱人,也很幼稚。

真实世界里的"干活",从来不是单纯的智力问题。

一个新人进公司,不是因为聪明就能立刻产出。他要知道项目背景,知道权限边界,知道谁能拍板,知道代码怎么跑,知道测试怎么验,知道哪个东西不能碰,知道历史上踩过什么坑。

**工作不是智力的裸奔。工作是一整套被环境驯化过的行为。**

AI Agent 也一样。模型本身只是一颗高潜能的大脑。真正决定它能不能干活的,是它周围有没有一套能把它组织成"劳动者"的系统。

这套系统,这两年慢慢有了一个名字:**Harness**。

这篇文章想做的事情,是把这套系统是怎么一步步长出来的讲清楚。我会沿着 prompt → tool use / ReAct / Toolformer → AutoGPT → coding agent → harness 的演化脉络往前走,然后回过头来回答三个问题:Harness 到底由什么组成?它和 Agent Framework 有什么区别?为什么它会成为 AI Agent 时代真正的护城河?

---

## 第一部分:Agent 运行层是怎么长出来的

### 一、为什么 Agent 一定会需要运行层?

因为 Agent 面对的不是"回答问题",而是"完成任务"。

这两件事差别巨大。

回答问题可以一次性完成,完成任务必然是长链路。

回答问题主要靠知识,完成任务必须靠状态、工具、反馈、验证、权限和协作。

回答问题错了最多重答,完成任务错了可能改坏代码、发错邮件、删错文件、污染数据库、误导业务决策。

这就是 Agent 从聊天机器人里分裂出来的根本原因。聊天机器人面对的是语言场,语言场里最重要的是"生成";Agent 面对的是行动场,行动场里最重要的是"约束"。

由此可以推出一个反直觉但很关键的结论:

**模型能力越强,运行层越不能薄。**

弱模型只能胡说,出不了大事。强模型可以真做事,一旦它能真做事,权限、沙箱、审计、回滚、验证、人类介入,就全部变成刚需。

这里也可以解释一句近来流行的说法:模型裸奔的时代结束了。

所谓"裸奔",不是说模型没有工具,也不是说回答得不好,而是说模型在没有明确任务边界、没有权限控制、没有外部验证、没有状态账本、没有失败归因、也没有人类接管机制的情况下,直接开始行动。它可以写文件,可以跑命令,可以调接口,可以给出看起来完整的结论,但系统并不知道它为什么这么做、做到了哪里、错了谁发现、失败了怎么停。

这才叫裸奔。

Harness 要解决的,就是让模型不再以一个自由漂浮的聊天框进入真实工作,而是以一个被运行层包裹的执行单元进入真实工作。

### 二、Prompt Engineering:我们以为问题在"怎么问"

最早大家对大模型的理解很简单:它像一个会写东西的人,你问得好,它就答得好。

于是 prompt engineering 成了第一门显学。最重要的问题是:怎么写清角色,怎么给示例,怎么拆步骤,怎么让模型按格式输出。

Prompt engineering 当然有价值。它教会了我们一件事:模型不是搜索引擎,而是一种可以被语言条件化的生成系统。

但 prompt 的边界也很明显。

它本质上只是一次性指令。没有真正的环境,没有稳定的状态,没有执行反馈,没有权限边界,没有任务账本,没有可验证的完成标准。

你让模型"请仔细检查代码",它会说自己检查了。但它到底跑没跑测试?有没有读相关文件?有没有理解依赖?有没有覆盖边界条件?——不知道。

这就是 prompt 最大的问题:

**它可以制造"认真工作的语言表象",但不能保证真实工作发生。**

适合"回答",不适合"执行"。

### 三、ReAct 和 Toolformer:模型从"说"走向"做"

Agent 的真正起点不是 AutoGPT,更早的转折发生在 ReAct 和 Toolformer。

ReAct 的核心非常简单,但意义很大:它让语言模型把推理和行动交错起来——不是先想完再输出,而是一边推理,一边采取动作,一边根据外部反馈更新计划。ReAct 论文明确说,reasoning traces 帮助模型跟踪和更新行动计划,而 actions 让模型能够接入外部知识库或环境获取信息。[^react]

这一步很关键。它第一次让模型从"闭门造车"变成"边看边做"。

以前模型回答问题,像一个人坐在屋子里凭记忆写报告。ReAct 之后,模型开始像一个实习生:先想一下,然后去查资料,拿到结果,再修正判断。

Toolformer 从另一个角度推进了这件事。它让模型学会什么时候调用 API、调用什么 API、传什么参数、如何把工具结果纳入后续预测。它测试的工具包括计算器、问答系统、搜索引擎、翻译系统和日历等。[^toolformer]

这两条路线合起来揭穿了一个事实:

**模型不是所有能力都应该自己内部解决。**

算术不必硬算,检索不必硬背,日历不必凭空猜。翻译、搜索、数据库查询、代码执行,都可以交给工具。

于是 Agent 的问题从"模型会不会"变成了:模型什么时候该用工具?工具结果怎么返回?工具失败怎么办?工具输出能不能信?工具调用有没有权限?调用过程有没有日志?

注意,这时运行层已经出现了,只是还很薄——通常只是一个工具调用协议、一个函数 schema、一个 action-observation 循环。

但方向已经变了。模型不再只是生成文本,而是开始被嵌入一个外部行动系统。

### 四、AutoGPT 的失败:自动循环不等于可靠执行

2023 年前后,AutoGPT、BabyAGI 这些早期 autonomous agent 火了一阵。

这波浪潮特别重要。不是因为它们真正好用,恰恰相反,是因为它们把自动化 Agent 的问题暴露得非常彻底。

当时的想象很浪漫:给 AI 一个目标,它自己拆任务、自己搜索、自己写文件、自己反思、自己继续执行,最后交付结果。

但真实体验往往是:目标跑着跑着就漂了,搜索了一堆资料却不知道取舍,反思看起来很认真但下一步还是错,循环越来越长,token 越烧越多,中间状态乱掉,最后生成一个看似完成、其实没验收的东西。

这时候大家才发现:

**自动循环不等于可靠执行。**

会自己跑,不代表知道自己在跑什么。会反思,不代表能归因失败。会调工具,不代表能安全完成任务。

AutoGPT 这一代 Agent 最大的历史意义,是它用失败证明:Agent 不需要更多自我意识式的大词,Agent 需要的是更扎实的运行结构。需要计划,但计划要能被约束。需要记忆,但记忆要能被管理。需要工具,但工具要能被授权。需要反馈,但反馈要能进入状态。需要自主,但自主必须带着验收标准。

打个比方:这就像公司招了一个极聪明、极兴奋、极主动的新员工。他什么都想干,到处点开系统,随手改文档,自己给自己定目标,日报写得很好看。但你问他:任务编号是什么?需求是谁确认的?改了哪些文件?测试跑没跑?失败原因是什么?是否影响线上?谁 review?怎么回滚?——他答不上来。

那他不是生产力,是风险源。

AutoGPT 的失败,本质上不是模型失败,而是**运行层缺位**。

更直白地说:**没有状态管理、没有验证、没有权限、没有失败归因的 autonomous agent,只是在自动化地制造不确定性。**

### 五、Coding Agent:运行层真正成型的战场

为什么 coding agent 是 Agent 运行层成型的战场?

因为写代码这件事非常残酷。它不像写文章,语气顺、结构完整就能交差。代码任务要面对真实环境:代码库很大,依赖复杂,文件互相影响,测试可能失败,类型检查可能报错,已有功能不能破坏,改动要能被 review,PR 要能解释,bug 要能复现,修复要能验证。

这时,模型再会写代码也不够。因为真正的问题不是"能不能写出一个函数",而是:

**能不能在一个已有工程系统里做出可验证、可维护、可回滚的改变。**

SWE-agent 把这个问题讲得很清楚。它提出 Agent-Computer Interface(ACI),通过设计适合语言模型的命令和反馈格式,让模型更容易浏览仓库、查看文件、编辑代码、执行程序;好的 ACI 设计会显著提升 agent 表现。[^swe-aci]

这句话非常重要。它说明 Agent 能力不只来自模型,还来自接口。

同一个模型,给它原始 shell,它可能很笨;给它专门设计过的文件查看器、编辑器、搜索命令、lint 反馈,它突然就像"更聪明"了。

但模型真的变聪明了吗?没有。模型没变,变的是环境的暴露方式。

这就是 Harness 思维的起点:

**能力不是模型单独拥有的。能力是在"模型—接口—环境—反馈"这个系统里涌现出来的。**

Claude Code 也是这个方向。它不是一次性给建议,而是在工具循环里读取、修改、执行、观察,再继续下一步。

OpenAI Codex 的云端形态更明显。OpenAI 介绍 Codex 时说,它是云端软件工程 agent,可以并行处理多个任务;每个任务运行在自己的云端 sandbox 环境里,预加载用户代码库;它能读写文件、运行测试 harness、linters 和 type checkers。[^codex]

这已经不是"聊天框写代码"了,而是一个简化版工程现场:有仓库,有隔离环境,有命令执行,有测试,有 PR,有并行任务,有人类 review。

换句话说,coding agent 把 Agent 从"语言产品"逼成了"运行系统"。它必须有工位、工具、权限、日志、验收。

所以 coding agent 是 Agent 史上的关键分水岭。在它之前,Agent 更像一个聪明助手;在它之后,Agent 开始像一个受控执行单元。

### 六、举个 coding agent 翻车的例子

抽象的话不够直观,讲一个常见场景。

用户说:"帮我把登录页的小问题修一下。"

如果没有 Harness,agent 很可能先搜索全仓库,看到一个相似组件,顺手重构一片表单逻辑;它改了三四个文件,解释说"已经统一了登录体验",但没有跑项目里的测试,也没有打开浏览器确认登录流程,更没有注意到这个仓库里有一条隐含规则:认证相关代码不能在没有 review 的情况下顺手抽象,因为上一次抽象就引入过 session 过期 bug。

最后它给你的是一段看起来合理的 diff,而不是一次可验收的修复。

同一个任务,如果有 Harness,路径会完全不同:先从 issue 或用户描述里收窄目标,只改登录页相关文件;读取 AGENTS.md 里的测试命令和禁改目录;修改后跑对应测试和 lint;用浏览器点一次登录流程;最后把 diff 限制在可 review 的范围内。

模型可能还是同一个模型。区别是,它被放进了一个知道事实来源、权限边界、验证方式和历史坑位的工作环境里。

这就是 Harness 真正的作用。

## 第二部分:Harness 是什么

### 七、Harness:模型外面那层东西终于被命名

到了 2026 年,AI Harness Engineering 这个词开始变得非常准确。

一篇 2026 年的论文把它定义为围绕 foundation-model software agent 的 runtime substrate,负责 task specification、context selection、tool access、project memory、task state、observability、failure attribution、verification、permissions、entropy auditing、intervention recording 等职责。作者还明确说,软件工程能力不是模型单独产生的,而是 model–harness–environment 系统的涌现属性。[^harness-paper]

这个定义有穿透力。它把过去几年所有碎片化实践串了起来:

- Prompt engineering 是 context 的早期形式
- Function calling 是 tool access 的早期形式
- AutoGPT 的任务循环是 task state 的粗糙尝试
- SWE-agent 的 ACI 是 interface harness
- Claude Code / Codex 是 coding runtime
- MCP 是工具和资源接入协议
- Skills 是能力封装和上下文渐进加载
- AgentKit 是平台化 control plane
- GitHub Agent HQ 是把 agent 放进真实协作流程

这些看似不同的东西,其实都在补同一个洞:

**模型自己没有工作现场,Harness 给它造工作现场。**

由此可以收口出一个判断:

**Agent 的本质,不是"一个模型加工具",而是"模型被运行层组织成可执行行为"。**

这个判断成立的话,我们看 Agent 的方式就要变。以后评估 Agent,不能只问"用的什么模型、上下文窗口多大、benchmark 几分、支持多少工具",更要问:任务状态怎么保存?上下文怎么选择?工具权限怎么分级?执行轨迹怎么记录?失败怎么归因?验证信号从哪里来?人类如何介入?经验如何沉淀?多 agent 如何协作?系统如何防止熵增?

这才是 Agent 工程的真正问题。

### 八、Harness 的七个组成部分

如果把 Agent 运行层拆开,它至少有七个核心模块。

我列这七个,不是为了堆概念,而是为了说明一件事:**为什么 prompt 不可能承担这些职责。**

#### 1. 任务规格层

用户会说"帮我优化一下这个项目""研究一下这个行业""把这个功能做了""帮我写个报告"。这些话,对人类来说也不够明确,更别说 Agent。

运行层第一件事,是把自然语言愿望翻译成任务规格:什么叫完成,约束是什么,不能做什么,优先级是什么,输出形式是什么,是否允许联网,是否允许改文件,是否需要人类确认,验收标准是什么。

没有任务规格,Agent 就会自作主张。它会把"帮我看看"理解成"我可以大改一通",把"优化"理解成"重构",把"研究"理解成"泛泛总结",把"写长文"理解成"堆资料"。

问题不在模型不聪明,在任务没有被压成一个可执行对象。

#### 2. 上下文选择层

很多人对上下文管理有一个误区:窗口越大,Agent 越强。

不完全对。

上下文不是仓库,而是注意力预算。把所有东西都塞进去,模型不一定更聪明,反而可能更混乱。

真正的 context engineering 不是"多塞",而是:哪些信息常驻,哪些信息按需读取,哪些信息只在特定任务出现,哪些历史已经过期,哪些规则优先级更高,哪些上下文会造成误导。

Claude Skills 的 progressive disclosure 是一个典型例子。Anthropic 解释说,Skills 是包含 instructions、scripts、resources 的文件夹,agent 可以动态发现和加载;拥有文件系统和代码执行工具的 agent 不需要一开始就把整个 skill 读入上下文,所以一个 skill 能打包的上下文实际上可以非常大。[^anthropic-skills]

背后的思想很关键:**上下文要分层暴露**。先让 Agent 知道有什么能力,相关时再读怎么做,必要时再深入资源文件、脚本、模板、示例。

AGENTS.md 也是同样的逻辑。它像给 agent 的 README,是一个专门、可预测的位置,用来提供项目上下文和指令。

这件事看似很小,其实很大——它代表软件项目开始给 Agent 写"入职手册"。README 是给人看的,AGENTS.md 是给 agent 看的。CLAUDE.md、SKILL.md、project rules、recipes,本质上都是同一种东西:

**把原本藏在人脑、团队习惯和 Slack 历史里的隐性知识,变成 Agent 可读取的运行资产。**

#### 3. 工具访问层

真正危险的不是模型胡说,而是模型能动手。

模型只能输出文字时,风险主要是误导。模型能调用工具时,风险就变成行动事故。

MCP 官方规范把 tools 定义为服务器暴露给语言模型调用的能力,可以用于查询数据库、调用 API 或执行计算;resources 则是服务器向客户端暴露的上下文数据,比如文件、数据库 schema 或应用特定信息。[^mcp-tools][^mcp-resources]

外部世界正在被标准化地接入模型。这是巨大进步,但进步的另一面是风险。

工具越多,Agent 的身体越强;身体越强,破坏半径越大。

只会搜索的 Agent 最多查错资料;能写文件的 Agent 可能改坏代码;能跑 shell 的 Agent 可能删错目录;能访问数据库的 Agent 可能泄露数据;能发邮件的 Agent 可能代表你对外承诺;能调支付、CRM、工单系统的 Agent,已经不是助手,是业务执行者。

所以工具访问层的核心不是"接入更多工具",而是:哪些工具默认不可见,哪些工具只读,哪些工具可写,哪些工具要人工确认,哪些参数要校验,哪些输出要过滤,哪些工具调用要记录,哪些动作可回滚,哪些动作绝对禁止。

这是 Agent 工程里很残酷的一点:

**你不能只为成功路径设计 Agent。你必须为失败、误用、越权和失控设计 Agent。**

#### 4. 状态管理层

聊天可以无状态,任务不能。

一个 Agent 真正干活时,必须知道:现在任务进行到哪一步,前面做过什么,哪些尝试失败了,哪些假设被推翻了,哪些文件被修改了,哪些测试还没跑,哪些地方需要人类确认,哪些输出已经交付,哪些风险还挂着。

这就是 task state。

没有 task state,Agent 就会像失忆的外包。每轮都很聪明,连起来很蠢:它会重复搜索同一件事;会忘记自己刚刚失败过;会换一种说法继续犯同一个错;会不知道用户已经批准了哪一步;会不知道哪些约束不能再碰。

所以运行层必须把任务状态从聊天文本里抽出来。

聊天记录是叙事,任务状态是账本。叙事给人看,账本给系统跑。

#### 5. 验证层

这是最重要的一层,也是很多 Agent demo 最大的骗局所在。

模型很擅长说:"已经完成。""我已经检查。""修复成功。""以下是最终方案。""这个实现是稳定的。"

但它说完成,不代表真的完成。

真实完成必须有外部信号:代码任务要跑测试,数据任务要校验 schema,研究任务要有来源和交叉验证,文档任务要符合格式,客服任务要用户问题解决,销售任务要 CRM 状态变化,运营任务要系统指标变化。

没有验证层,Agent 就会把"生成了一个结果"误认为"完成了一个任务"。

这就是为什么 AgentKit、trace grading、evals、observability 开始变成主角。OpenAI 发布 AgentKit 时,把 Agent Builder、Connector Registry、ChatKit,以及 datasets、trace grading、automated prompt optimization 等 eval 能力放在同一组产品里,说明行业已经从"能不能跑"进入"怎么证明它跑对了"。[^agentkit]

Agent 的可靠性,不是靠它语气自信,而是靠执行轨迹、验证信号和可复盘记录。

一句话:**Agent 不能自证清白,它必须被系统验证。**

#### 6. 观测与归因层

生产系统最怕黑箱,Agent 尤其如此。因为 Agent 的错误往往不是单点错误,而是链式错误。

它可能第一步检索就错了;第二步基于错资料做计划;第三步调用了不合适的工具;第四步工具返回失败,它误读了;第五步为了完成任务开始编;最后输出一个很完整的错误结果。

只看最终答案,很难知道错在哪。

所以必须记录过程:它看了什么上下文,调用了什么工具,工具返回了什么,它如何解释返回结果,它在哪一步改变计划,失败后如何归因,有没有请求人类介入,最终验证是否通过。

这就是可观测性。

没有可观测性,Agent 一旦出错,团队只能吵架:是模型不行?是 prompt 不行?是工具坏了?是上下文错了?是权限不够?是测试不完整?是用户任务没说清?

没有 trace,所有人都在猜;有 trace,才能归因;归因之后,才能改系统。

Harness 的真正价值在这里露出来了:

**它不仅让 Agent 做事,还让 Agent 的失败变得可学习。**

失败不可怕,不可归因的失败才可怕。

#### 7. 人类介入层

很多人一谈 Agent,就喜欢讲 autonomous,仿佛越无人越高级。

这又是一个误区。

真实生产系统里,最重要的不是"完全自动",而是"在正确的地方自动,在正确的地方停下"。

什么地方要停?高风险动作前要停,不可逆动作前要停,权限不足时要停,上下文冲突时要停,验证失败时要停,用户目标不明确时要停,成本失控时要停,安全策略触发时要停。

成熟的 Agent 不是一个莽撞的自动机,而更像一个知道自己边界的员工。它可以自己完成低风险、可验证、可回滚的动作;但遇到高风险、不确定、不可逆的事情,它会升级给人类。

GitHub Agent HQ 的设计也体现了这个方向。它把多个 coding agent 放进 GitHub、GitHub Mobile、VS Code 等协作流里,让 agent 的输出以 comments、drafts、proposed changes、PR 等 artifact 形式进入 review 流程,而不是要求人类盲目接受。[^agent-hq]

这就是人机协作的真实形态。不是人类完全退出,而是人类从"逐字操作员"变成"边界设定者、审查者、裁决者"。

Agent 负责执行,人类负责方向、边界、风险和最终责任。

### 九、Harness 不是多 agent

讲到这里要澄清一个常见误会。

很多人一听 agent workflow,就自然想到 multi-agent。这很容易误导。

多个 agent 只是 Harness 可能使用的一种结构,不是 Harness 本身。

如果没有外部反馈,多 agent 可能只是把同一种错误并行放大。如果没有状态管理,多 agent 会互相覆盖、重复劳动、争抢上下文。如果没有清晰分工,一个 agent 同时当 planner、coder、reviewer、operator,会出现职责过载;多个 agent 角色边界不清,只是把这个问题放大。

所以判断一个 agent 系统的质量,不该先问"有几个 agent",而该先问:外部反馈在哪里?验收标准在哪里?状态如何交接?权限边界在哪里?错误如何被记录?有效经验如何回写?

这些问题答不清楚,再多 agent 也只是更复杂的 demo。

**Harness 的重点不是任务编排,而是反馈编排。**

### 十、Agent Framework 和 Harness 不是一回事

这里也必须讲清楚,因为很多人把 LangChain、AutoGen、CrewAI、OpenAI Agents SDK、Claude Code、Goose、OpenHands、Codex、MCP、Skills 全混在一起。

Agent framework 通常解决的是:如何定义 agent,如何编排步骤,如何调用模型,如何调用工具,如何让多个 agent handoff,如何构造 workflow。

Harness 解决的是更底层、更脏、更生产的问题:Agent 在哪里运行,能看到什么,能碰什么,怎么保存状态,怎么验证结果,怎么记录过程,怎么处理失败,怎么限制权限,怎么让人类介入,怎么让经验沉淀,怎么让整个系统可维护。

更短一点说:**Framework 像剧本,Harness 像片场。**

剧本重要,但没
有场务、灯光、道具、安全、调度、预算、监制,电影拍不出来。

同理,没有 harness,Agent framework 很容易变成漂亮 demo——它可以跑,但不可控、不可审计、不可复盘、不可规模化。

这就是为什么 Agent 的竞争会从"谁会编排"转向"谁能治理"。

### 十一、不同产品把运行边界放在不同位置

很多 agent 产品看起来都叫 Agent,但它们真正不同的地方,不是 UI,不是模型,不是名字,而是:

**它把运行边界放在哪里,治理边界又放在哪里。**

这里有两个轴。一个是物理运行边界:Agent 到底在模型内部、本地机器、云端 sandbox,还是协作平台里运行。另一个是治理边界:谁决定它能看什么、能碰什么、什么时候停、结果怎么算完成、失败怎么归因。

边界位置决定产品形态,也决定风险结构。

**第一种,边界放在模型内部**(Toolformer 路线)。让模型自己学会什么时候调用工具、传什么参数、如何吸收结果。优点是自然流畅,用户感觉不到太多外部系统存在。问题是:模型内部能力不等于生产治理能力。模型知道可以调用日历,不代表它该有权限改你的日程;知道可以查数据库,不代表它该看到所有字段。模型内部解决的是"能不能",Harness 解决的是"该不该、怎么做、出事怎么办"。

**第二种,边界放在项目文件里**(AGENTS.md / CLAUDE.md / SKILL.md / project rules)。把团队知识写成文件,让 Agent 在任务开始时读取。优点是务实,不要求重构整个平台,加一个文件就能让 Agent 更懂这个项目。但文件说明不能替代运行控制——它告诉 Agent 跑测试用什么命令、不要改哪些文件,但不能强制执行所有规则。真正的强制执行还得靠权限、hook、sandbox、CI、review、policy engine。所以项目文件是 harness 的知识层,不是完整 harness。

**第三种,边界放在本地客户端**(Claude Code、Codex CLI、Goose、OpenClaw)。优点是离真实工作环境近:能读本地仓库、跑命令、接编辑器、改文件,反馈快、交互自然。但风险也明显:它太靠近你的机器,权限给大了就可能动到不该动的东西;命令执行不受控,就可能造成破坏;本地 secrets、配置文件、私有代码都可能暴露在工具范围里。本地 agent 的本质不是"更方便的聊天",而是一个带权限的执行器——本地越强,越需要保险丝。

**第四种,边界放在云端 sandbox**(Codex Web 是典型)。把任务放进云端隔离环境,每个任务独立运行,预加载代码库,可以读写文件、跑测试、提交 PR。好处是隔离性强、可异步执行、可并行任务、更容易记录过程、更适合企业审批和 PR 化交付。问题是环境还原成本高:私有依赖怎么装、secrets 怎么管理、网络权限怎么开、本地特殊环境怎么模拟、任务和真实生产系统之间怎么保持一致。可以把它想成一个远程外包工程师——你给他仓库、任务、环境、权限,他在隔离房间里干活,最后给你 PR。比让他坐在你电脑前安全,但沟通和环境配置成本更高。

**第五种,边界放在协作平台**(GitHub Agent HQ)。它不是单独做一个 agent,而是把多个 agent 放进 GitHub 这个协作现场。这一步很关键,因为真正的软件工程不是在聊天框里发生的,而是在 issue、branch、PR、review、CI、release、discussion 里发生的。一旦 Agent 进入协作平台,它就要接受团队治理:谁能指派、谁能审批、谁能 merge、谁看日志、谁对结果负责、多个 agent 结果冲突怎么办、agent 产出的 PR 如何标识、是否消耗预算、是否符合公司安全策略。这就是 Agent 的 control plane。

这五种不是同一个维度上的五个抽屉,更准确地说,它们是"运行位置 × 治理位置"组合出来的五个常见产品落点。

模型内部边界最顺滑,但治理最弱;项目文件边界最轻量,但只能指导;本地客户端最贴近现场,但风险贴近机器;云端 sandbox 更适合异步交付,但环境复现更难;协作平台最接近组织控制面,但需要更重的流程设计。

判断 agent 产品时,不要只看 demo 多顺。要看它的运行边界放在哪里,以及这个边界能不能承载目标任务的风险、状态、验证和协作需求。

## 第三部分:Harness 的长期意义和实践启发

### 十二、最深的一层:Harness 是把组织知识显性化

只从技术层面讲 Harness 还不够深。因为 Harness 不只是技术运行时,它更像一种组织知识的外骨骼。

为什么这么说?

一个成熟员工为什么比新人强?不是因为智商更高,而是因为他知道很多隐性东西:这个项目历史上为什么这样设计;哪个模块不能轻易改;哪套测试最关键;哪个客户最敏感;哪个指标不能牺牲;哪个需求看似简单其实有坑;哪个同事说"可以"其实还要二次确认;哪个系统文档已经过期;哪个脚本跑起来会污染数据;哪个数据库字段名字很误导。

这些东西在组织里到处都是,但通常不在正式文档里。它们存在于老员工脑子里、Slack 记录里、PR 评论里、事故复盘里、代码注释里、测试命名里、需求会议里、没人敢删的脚本里、"这个千万别动"的口头提醒里。

人类新人要靠时间慢慢社会化,Agent 没有这个过程。

所以 Harness 的一个深层功能,就是把组织里的隐性工作知识转成机器可使用的结构。AGENTS.md 是一种,Skills 是一种,Recipes 是一种,MCP resources 是一种,tracing + failure attribution 是一种,Runbook 是一种,policy engine 是一种,eval datasets 也是一种。

这件事的重点不是"把知识存起来"——普通知识管理经常停在保存。Harness 要更进一步:

**把知识放到行动发生的位置,让它能改变 Agent 的选择、权限、验证和停止条件。**

如果一个历史坑只存在于某个老员工脑子里,Agent 当然不会知道。如果它只存在于三个月前的一条 Slack 消息里,Agent 也很难稳定用上。如果它变成 AGENTS.md 里的禁改说明、变成 skill 里的检查步骤、变成 CI 里的测试、变成权限策略里的人工确认点——Agent 才真的能在工作时碰到它。

所以可以推出一个更大的判断:

**Agent 工程的下一阶段,不是让模型更像人,而是把组织经验变成 Agent 可执行的环境。**

这件事意味着,企业真正沉淀的资产,不只是代码和数据,还包括任务模板、工作流程、验收标准、权限模型、失败案例、上下文索引、工具接口、审查策略、自动化回归测试、人类介入规则。

这些东西会形成组织的 Agent 运行层。

模型可以换,但运行层会越用越厚。**谁的 Harness 越厚,谁越能吃模型升级红利。** 没有 Harness 的团队,每次模型升级都要重新写 prompt、重新试错、重新踩坑。有 Harness 的团队,模型升级更像更换发动机——底盘、刹车、仪表盘、驾驶规则还在。

### 十三、为什么竞争点会从 Model 转向 Runtime

不是说模型不重要。模型当然重要,更强的模型会提高上限。

但生产系统的价值不只看上限,更看稳定性、复用性、可控性、集成成本和风险边界。

模型像发动机,Harness 像整辆车。发动机再强,没有刹车、方向盘、仪表盘、安全带、底盘、变速箱、道路规则,你不敢上路。

AI 也一样。一个企业真正部署 Agent 时,不会只问"哪个模型最强",还会问:怎么接内部系统?怎么限制数据访问?怎么做审计?怎么评估效果?怎么处理越权?怎么记录成本?怎么接工单?怎么进入审批流?怎么把结果给业务系统?怎么持续优化?

这些问题一旦解完,就会沉淀成 runtime advantage。

模型升级时,强 runtime 可以直接吃红利;没有 runtime 的公司,每次模型升级都要重新试、重新训人、重新写 prompt、重新踩坑。

这就是运行层复利。

今天很多人还在比模型,但真正做生产的人已经开始比"模型周围的系统":谁的 context routing 更准,谁的 tool governance 更稳,谁的 sandbox 更安全,谁的 observability 更细,谁的 eval 更贴近业务,谁的 human-in-the-loop 更自然,谁的 skills / recipes / project memory 更可复用,谁能把 Agent 嵌入真实工作流而不是困在聊天框里。

这才是下一阶段的护城河。

### 十四、对软件工程的影响

软件工程会先被改变,因为它最适合 Agent。

原因很简单:代码是文本;测试可自动化;版本控制成熟;PR 流程清晰;失败容易复现;产出可以被 diff;review 机制已有;CI/CD 已经是天然验证层。

所以 coding agent 是第一块成熟战场。

但它改变的不只是写代码速度,更深的变化是工程组织方式。

以前一个工程师的主要工作是理解需求、写代码、调试、跑测试、发 PR、处理 review。未来很多低到中等复杂度的任务会被 agent 执行。工程师的重心会迁移到:拆任务、定义边界、写更好的 AGENTS.md、维护测试、设计架构、审查 Agent 输出、组织多个 Agent 并行探索、做风险裁决、沉淀项目知识、维护 harness。

这不是"程序员消失",而是程序员的工作抽象层提高。

以前你亲手拧螺丝,后来你操作机器,再后来你设计产线。Agent 时代,工程师越来越像一个小型研发组织的调度者。

### 十五、对 agent 产品的启发

理解 Harness 后,再看 agent 产品,判断标准会变得很不一样。

很多 agent 产品看起来是在卖"更聪明的助手",但真正的壁垒可能不是模型调用,而是默认 Harness 设计:

它有没有理解用户的真实工作环境?有没有把任务边界设计清楚?有没有接入正确工具?有没有可观测状态?有没有可靠验证?有没有失败恢复?有没有把用户反馈转成长期能力?有没有把高频工作流打包成可复用资产?

这些问题,比"用了哪个模型"更接近产品本质。

因为模型会持续变强,工具接口会持续变化,但一个领域里的工作流、约束、验收方式和组织经验,才是更长期的复利来源。

未来很多 agent 产品的差异,可能不在"谁能调用模型",而在"谁更懂某类任务应该被怎样托管、验证和改进"。

### 十六、Harness 不是越厚越好

讲到这里,容易走向另一个误区:既然 Harness 很重要,是不是越复杂越好?

不是。**Harness 的厚度要和任务复杂度匹配。**

简单任务不需要重型 Harness。让模型改一个小文案、解释一段代码、生成一个临时脚本,薄一点的约束就够了。

复杂任务才需要更强的 Harness。比如跨文件重构、长时间自动开发、多角色协作、生产环境操作、金融决策、研究工作流、持续维护知识库。这类任务里,真正危险的不是模型不会生成,而是它在长链路中缺少反馈、状态、权限和验收。

所以"Big Model vs Big Harness"不是一个真正的二选一问题。更好的问法是:

**这个任务是否已经超过单次生成的可靠边界?**

如果没有,薄 Harness 足够;如果超过了,就必须把验证、约束、状态和回写认真设计出来。

### 十七、对个人意味着什么

对个人使用者来说,Harness 的启发也很直接:

不要只问"我该怎么提示 AI",要开始问:

- 我的工作事实来源在哪里?
- 我的项目规则有没有写下来?
- 哪些任务可以被稳定交给 agent?
- 我怎么验收 agent 的结果?
- 我每次修改 agent 输出的原因,有没有沉淀成规则?
- 哪些重复流程可以变成 skill?
- 哪些知识可以变成 agent 可用的 context infrastructure?

你会发现,真正提高 AI 使用水平的,不是收藏更多 prompt,而是把自己的工作环境整理成 agent 能读、能用、能验证、能回写的系统。

这也是 AI coding 给普通人的最大启发:

**以后重要的能力,不只是"会不会和 AI 聊天",而是"会不会搭一套让 AI 真能工作的环境"。**

### 十八、Agent 的真正瓶颈不是智力,而是信任

讲到最后,必须回到一个词:**信任**。

为什么很多 Agent demo 看起来很强,但企业还不敢大规模放权?不是因为它们完全不能干活,而是因为人类不信任它们。

而信任不是靠宣传建立的,信任来自几个东西:行为边界清晰,过程可观察,失败可归因,结果可验证,权限可控制,责任可追溯,经验可复用,成本可预测。

这些东西,都不是模型本身给你的,它们来自 Harness。

所以 Harness 的本质,是**信任基础设施**。

没有它,Agent 只能停留在"你帮我试试";有了它,Agent 才可能进入"你负责这块"。

这两个状态差别巨大。"你帮我试试",人类还是主操作者;"你负责这块",Agent 才真的变成生产单元。

要从前者走到后者,靠的不是更会聊天,靠的是运行层。

### 十九、Harness Engineering:Agent 时代真正的工程学

过去的软件工程,核心对象是代码;后来的平台工程,核心对象是服务;DevOps 的核心对象是交付流水线;MLOps 的核心对象是模型生命周期。

Agent 时代,核心对象会变成:**智能行为的运行过程。**

这个过程既不是传统程序,也不是纯模型推理。它介于两者之间——有概率性,有状态,会调用工具,会根据反馈调整,会产生中间产物,会受到上下文影响,会在权限边界内行动,会和人类协作,会出错,而且错误形态复杂。

所以它需要新的工程学。

这门工程学不只是 prompt engineering,也不只是 agent framework,更不只是模型微调。它是:**Harness Engineering。**

它研究的是:如何把模型能力接入环境;如何把环境暴露给模型;如何把任务变成状态机;如何把工具变成受控接口;如何把上下文变成可加载资产;如何把执行变成可观测轨迹;如何把失败变成可归因数据;如何把结果变成可验证交付;如何把人类介入变成制度,而不是临时救火。

这才是深水区。

### 结尾:模型裸奔的时代快结束了

回到开头那句判断:**AI Agent 的历史,不是模型越来越聪明的历史,而是人类不断给模型补上工作现场的历史。**

一开始,我们只有 prompt。后来有 tool use,再后来有 ReAct 循环,再后来有 autonomous agent,再后来有 coding agent。现在开始出现 skills、AGENTS.md、MCP、sandbox、trace、eval、AgentKit、Agent HQ、project memory、permission modes。

这些东西不是碎片,它们共同指向一个事实:

**模型裸奔的时代快结束了。**

未来真正能落地的 Agent,不会是一个自由漂浮的聊天框,它会被放进一个运行层里。这个运行层会告诉它:你是谁,你要做什么,你能看什么,你能碰什么,你怎么行动,你怎么验证,你什么时候停,你出了错怎么解释,你如何把经验留给下一次。

这听起来不像科幻,但这恰恰是 AI 进入真实世界的方式。

真实世界不奖励"看起来聪明",真实世界奖励"可交付、可复盘、可控制"。

所以最后一句话可以压得狠一点:

**Agent 的未来,不是让模型像神一样无所不能,而是把模型关进一套足够好的工作制度里,让它像一个可靠的劳动者一样产出。**

这就是 Harness 的意义,也是 AI Agent 从玩具走向生产力的分界线。

所以未来讨论 AI Agent,不该只问模型有多强,也不该只问有几个 agent。更关键的问题是:

**你给它建了一个怎样的工作环境?**
