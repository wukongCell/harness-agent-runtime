# 一念到交付之间

## 引子：不要急着给自己找护城河

最近围绕 MASL 和 agent runtime 的讨论，越来越逼近一个不太舒服但必须承认的问题：如果 Codex、Claude Code 这类 runtime 继续扩张，原生支持 `/goal`、plan、review、subagents、hooks、memory、dashboard、owner summary，MASL 还剩什么？

这个问题不能用“runtime 还没那么强”来糊弄过去。它现在还没那么强，不代表方向上不会吃过来。更不能用“我们有 phase-first control plane”来安慰自己，因为如果 phase、review、handoff、rerun、progress、summary 这些都只是通用 orchestration，那么 runtime 天然有更好的位置去做它：它掌握模型调用、thread state、工具链、用户入口、telemetry 和产品界面。

所以必须先把自我叙事拆掉。

MASL 没有比 LLM 更高级的智能。MASL 也不应该幻想自己能在通用 agent orchestration 上长期胜过 runtime。最底层仍然是：

```text
prompt/context -> LLM -> output/tool action
```

意图恢复是 LLM 做的。truth contract 是 LLM 生成或辅助生成的。执行 schema 是 LLM 根据上下文和约束临时组织出来的。review routing、owner summary、evidence explanation，本质上也大多是把信息送进 LLM，让它输出结构化判断、文本或工具调用。

这不是贬低 MASL，而是把它放回真实位置：MASL 不是另一个智能引擎。它最多是给智能引擎提供外部承载、稳定机制、恢复机制和人类价值判断的沉淀方式。

如果 agent runtime 能从用户原始 prompt 稳定恢复真实意图，临时生成可靠执行 schema，自己拆任务、调 agent、验收证据、关闭 review gap、向用户解释状态，并在失败时自我恢复，那么 MASL 作为独立控制面确实会被压缩，甚至失去必要性。

这篇文章不是为了证明 MASL 必然有护城河。它要回答一个更诚实的问题：在 runtime 正在变强、LLM 作为核心引擎持续进化的今天，MASL 当下到底在探索什么？它做的这些机制有什么阶段意义？如果未来被 runtime 吃掉，它还能沉淀成什么？

## 一、最底层：原始 prompt 是低带宽意图信号

一切从用户一句话开始。

用户不会天然给出一个完整的机器可执行 spec。用户通常说的是：

```text
这个页面不太对，修一下。
让系统能自动回推。
我觉得这个方向不稳。
把这块做得用户能看懂。
不要只讲 what，要讲 why/how。
```

这些话里有目标，也有情绪；有要求，也有背景；有明确约束，也有未说出口的价值判断。用户给出的不是一个完整 JSON：

```json
{
  "goal": "...",
  "non_goals": [],
  "acceptance": [],
  "risk_policy": {},
  "evidence_required": [],
  "owner_actions": []
}
```

用户给的是低带宽意图信号。

LLM 的第一项能力，就是从这个低带宽信号里恢复高带宽任务真相：

```text
用户真正想要什么？
什么只是背景？
什么是硬约束？
什么是非目标？
什么风险不能擅自越过？
什么算完成？
什么需要追问？
什么可以合理推断？
```

如果 LLM 足够聪明，它可以临时生成一个很好的执行 schema。它可以说：我理解你的目标是 A，非目标是 B，成功条件是 C，风险边界是 D，我会先做 E，再用 F 验证，最后用 G 向你交付。

如果这个过程每次都稳定准确，那么外部系统需要做的事就会少很多。也就是说，MASL 的很多 intake、truth contract、phase schema，不是原则上不能被 LLM/runtime 做掉，而是当前阶段用外部结构提高稳定性。

所以我们不能把 truth contract 神化。truth contract 不是绝对真相，它也是 LLM 或人机协作生成的。它更准确的定义是：

```text
当前系统对用户意图、约束和完成标准的显式可检验假设。
```

它的价值不在于永远正确，而在于四点：

1. 显式化：让用户、agent、reviewer 都知道系统以为什么是真的。
2. 可修正：如果错了，可以指出、更新、重跑。
3. 可复用：同类需求下次不用从零猜。
4. 可验证：可以把 evidence 绑定到每条假设上。

这已经把 MASL 从“定义真相的系统”拉回了更谦逊的位置：它是帮助 LLM 外化、检验和修正 truth hypothesis 的系统。

## 二、LLM 是智能，工程化是稳定性放大器

如果承认最底层都是 prompt/context 到 LLM，那么工程化还有什么价值？

答案不是“更聪明”。工程化不能凭空创造比模型更高的语义智能。它的价值在于：

```text
稳健
复现
沉淀
扩展
恢复
```

稳健，是同类任务不要每次靠 LLM 临场悟性。  
复现，是这次为什么判定完成，下次别人能不能重放。  
沉淀，是用户价值判断、项目偏好、历史踩坑、失败模式不能每次丢进 prompt 重新赌。  
扩展，是 LLM 上下文有限，需要外部文字、文件、状态、schema、evidence ledger 扩展认知带宽。  
恢复，是单次不行，可以把中间状态保存下来，缩小下一次问题空间，恢复重试。

这和人类的认知工具很像。人类不是因为没有智能才发明语言、文字、纸张、账本、日历、测试、日志、代码评审、会议纪要。人类这么做，是因为工作记忆有限，复杂推理需要中间符号，多人协作需要共享状态，长期项目需要跨时间保存意图，错误需要被记录，标准需要被固定。

LLM 也是一样。即使模型很强，它仍然面对：

```text
上下文窗口有限
注意力会稀释
长期状态不稳定
工具观察需要外部化
多轮任务需要中间承载
错误模式需要反馈沉淀
```

所以 MASL 不应该被理解为替 LLM 思考。更准确地说，它是在给 LLM 提供纸张、账本、验收单、检查清单、证据柜、交接单和回头路。

这就是“外部认知结构”的真实含义。但这个词太硬，落到具体动作上就是：

```text
把用户的一念，逐步外化为可检验假设；
把执行中的失败，外化为下一轮修正信号；
把用户价值判断，外化为稳定偏好和规则；
把复杂任务，拆成足够小、可恢复、可验证的认知单元。
```

## 三、拆小不是形式主义，而是降低每次推理的智力负担

复杂任务难，不是因为每一步都难。很多时候是因为：

```text
目标太大
上下文太多
依赖太乱
中间状态没保存
错误反馈没定位
完成标准不清
```

如果把任务拆成足够小的单元，每个单元都有清楚输入、清楚输出、清楚验证、清楚失败原因和清楚下一步，那么每次 LLM 调用需要承担的认知负荷就会下降。

这才是 phase、subagent、handoff、runtime-progress、review loop 的真正意义。不是因为“多 agent”本身有神秘价值，而是因为它们可以降低单次推理调用所需的“临场天才程度”。

好的系统不是让 LLM 变聪明，而是让任务变得足够清楚，让标准智力水平也能完成。

这也解释了为什么持久化和恢复重要。单次执行不行，不代表系统失败。只要中间状态被保存，错误被定位，下一次上下文更好，系统就可以通过尝试、观察、记录、缩小问题、重试、校正来收敛。

人类工程实践也是这样。我们很少一次想对复杂系统。我们靠笔记、测试、日志、issue、回归测试、文档、代码评审和复盘，不断把不稳定的个人认知外化成可被团队复用的结构。

MASL 如果有价值，也是在做类似的事。

## 四、沉淀不是玄学，就是外部化、检索和注入

“沉淀”这个词容易被说玄。拆到底，它就是：

```text
让 LLM 把有复用价值的信息写到外部载体；
下次任务开始前，把相关信息取出来；
重新送回 LLM 上下文。
```

也就是：

```text
observe / decide / write down / retrieve / inject / act
```

写到哪里不是本质。可以是 Markdown、JSON、SQLite、Obsidian、Notion、vector store、git commit、test、schema、prompt rule。关键是两件事：

```text
未来能被检索到；
检索后能改变执行。
```

如果写了但未来 agent 不读，读了但不知道怎么用，用了但没有约束力，那就只是归档，不是真正的沉淀。

沉淀可以分成四类。

第一类是 Fact Memory，稳定事实。例如：本项目当前状态写 `PROJECT_CONTEXT.md`，不写 README；phase4/phase5 runtime progress 复用 `runtime-progress.json`；用户可见内容中文优先。

第二类是 Preference / Value Memory，用户和项目价值判断。例如：用户关心 AI 监工逐步退场；用户不想只讲 what，要讲 why/how 和第一性原理；用户认为叙事不是真相；用户希望 raw evidence 可以保留英文，但 owner-facing summary 要中文可读。

第三类是 Failure Mode Memory，错误模式。例如：review 打回后 rerun 可能只重跑验证，而不修 review gaps；LLM 容易把 narrative closure 当 completion；旧 completed runtime-progress 可能覆盖后续 completed facts；父进程旧模块可能继续运行新 phase。

第四类是 Policy / Contract Memory，已经固化的规则。例如：如果 phase 修改控制面文件，必须 phase-boundary safe-stop，让 fresh process resume；phase5 打回 rerun 时，必须注入 verdict / reason_codes / evidence_gaps。

沉淀真正有价值的时候，会从记录升级成可执行结构。路径通常是：

```text
Observation
  -> Finding
  -> Decision
  -> Policy
  -> Prompt rule / schema / test / code
```

例如：

```text
Observation: phase5 打回后，rerun phase4 可能只是重跑测试。
Finding: review feedback 没有成为 rerun 的中心上下文。
Decision: phase5 verdict / reason_codes / evidence_gaps 必须注入 rerun prompt。
Policy: 所有 review-derived rerun 都必须带上一轮 review feedback bundle。
Executable form: 测试断言 rerun phase4 prompt 包含 evidence_gaps；代码构造 feedback bundle；phase4 artifact 逐项回应 gap；phase5 逐项复审 gap closure。
```

这才是完整沉淀。

当然，沉淀有风险。因为沉淀通常也是 LLM 写的，错误也可能被写进去。最危险的是把假设写成事实。比如“MASL 的护城河是多进程 phase runner”，现在看就不应该沉淀成 fact。更准确的说法应该是：

```text
当前判断：多进程/phase runner 不是核心护城河；MASL 更可能是外部认知脚手架和 truth/policy/eval 沉淀层。此判断应随 runtime 能力演进继续修正。
```

所以有效记忆至少要区分：

```text
confirmed fact
working hypothesis
user preference
temporary decision
deprecated decision
failure mode
open question
```

沉淀不是越多越好。过期记忆、错误记忆、局部经验泛化、重复记忆，都会污染下一次上下文。沉淀系统最难的不是 append，而是筛选、结构化、检索、注入、过期管理和防止错误固化。

## 五、runtime 会吃掉 orchestration，这是事实

如果 MASL 的定义是：

```text
把需求拆成 design / implement / review；
启动多个 agent 或 session；
让它们交接上下文；
review 不过再打回。
```

那这个东西没有强护城河。

runtime 天然有更好的位置来做通用 orchestration：

1. 它拥有原始会话上下文：thread、tool calls、diff、terminal 输出、token usage、用户打断、compact 前后状态。
2. 它拥有 agent 启动能力：fork thread、spawn subagent、side conversation、工具权限。
3. 它拥有模型调用链路：可以在系统层注入 goal、review、continuation、budget，而不是靠外部 wrapper 拼 prompt。
4. 它拥有 UX 入口：用户已经在 Codex / Claude Code 里工作。它加 `/goal`、`/plan`、`/agents`、`/review` 比 MASL 要用户另开控制面自然。
5. 它有 telemetry 和训练闭环：能看到大量真实任务失败模式，把 orchestration prompt、tool policy、review loop 做进产品。

所以 generic multi-agent delivery orchestration 很容易被 runtime 吃掉。这不是悲观，而是结构事实。

MASL 当前很多能力也都可能被 runtime 原生化：

```text
phase3 / phase4 / phase5 -> /goal + plan + internal task phases
runtime-progress.json -> native progress events / dashboard
review loop -> /review + auto-fix loop
subagent handoff -> native subagents / side conversations
timeout observation -> native task budget / stall detection
owner-facing summary -> native delivery summary
context slicing -> runtime compact / memory / goal context
phase artifacts -> native task artifacts / tool logs
rerun prompt -> native continuation prompt with feedback
```

如果 MASL 长期围绕这些通用机制堆代码，方向是危险的。

真正的问题不是“我们能不能调度 agent”。LLM supervisor 会越来越会调度。它可以自己判断简单任务直接干，复杂任务拆成几个环节，派生子会话或子 agent，监督各个环节干活，然后整合结果。这确实就是 MASL 现在做的大部分事情。

所以 MASL 不能把“会分工”当成价值。

## 六、如果还有空间，空间在 truth recovery / correction / projection

更准确的分界，不是 runtime vs MASL，也不是 thread vs process，而是：

```text
LLM/runtime 是否能把低带宽 raw prompt 稳定恢复成高带宽 truth，
并在执行中持续用 evidence 和 feedback 校正这个 truth，
最后把 truth 外显给人。
```

这里有四个关键词：

```text
稳定
恢复
校正
外显
```

稳定，不是偶尔一次做得漂亮，而是同类任务反复做都不漂。  
恢复，是从用户原始 prompt 中恢复隐藏目标、约束、验收和风险边界。  
校正，是执行过程中发现证据不足、review failed、owner 改口时，能更新理解。  
外显，是让人类看见当前 truth，而不是只看最后一段总结。

如果 runtime 能做到这四个，MASL 就没了，或者至少会被压缩成 project policy / schema / eval pack。如果 runtime 做不到，MASL 的空间就在三件事：

```text
truth recovery
truth correction
truth projection
```

Truth recovery，是从 raw prompt 恢复用户真实意图、约束、验收、风险和非目标。  
Truth correction，是执行中用 evidence、review、owner action 持续修正这个 truth hypothesis。  
Truth projection，是把当前 truth 用人能快速判断和决策的方式外显出来。

这比“phase-first multi-agent control plane”更接近 MASL 的可能价值。

## 七、叙事不是真相

LLM 很擅长生成闭合叙事：

```text
我理解了问题。
我做了修改。
我跑了测试。
测试通过。
因此完成。
```

这个故事可能是真的，也可能只是 narrative closure。

真实交付需要区分几种闭合：

```text
故事闭合
证据闭合
review 闭合
owner 决策闭合
```

故事闭合只是语言上说得通。证据闭合要求某条 evidence 真的证明某条验收。review 闭合要求 review gap 被逐项处理或明确升级。owner 决策闭合要求用户需要做的取舍、授权或验收被明确记录。

很多 agent 系统的问题，不是不会做事，而是会把“看起来像完成”当成“真的完成”。尤其在长链路里，LLM supervisor 容易出现：

```text
忘记原始验收
将弱证据当强证据
子 agent 报告未验证
review feedback 没进入下一轮中心
final summary 过度闭合
owner action 被当成继续提示而不是状态迁移
```

如果 runtime 能稳定防住这些问题，MASL 价值下降。  
如果 runtime 只是收集 logs 然后让模型总结，MASL 仍有空间。

核心不是有没有证据，而是：

```text
系统能不能判断证据强度？
能不能把证据绑定到具体验收？
能不能承认证据不足时不能宣布完成？
```

## 八、LLM 临场自组织 vs 外化制度

LLM supervisor 的强项是临场智能：

```text
看到任务 -> 自己决定怎么拆 -> 自己调度 -> 自己修正
```

它灵活，低摩擦，不需要用户先适配系统。用户一句“这个体验不对，修一下”，它可以自己判断这是 UI polish，风险低，直接 inspect 页面、改代码、跑测试、给摘要。这比固定流程自然。

MASL 如果还有价值，不能靠比它更灵活，而要靠外化制度：

```text
把反复出现的正确性边界沉淀成 contract。
```

两者的差异是：

| 维度 | LLM supervisor 临场生成 | MASL 外化 contract |
|---|---|---|
| 灵活性 | 高 | 较低 |
| 启动成本 | 低 | 高 |
| 可审计 | 弱一些 | 强 |
| 可复现 | 弱一些 | 强 |
| 跨会话稳定 | 依赖 memory/summary | 依赖 durable state |
| 项目语义 | 需要每次理解 | 可固化 |
| owner 责任边界 | 容易被继续执行吞掉 | 可显式停机 |
| 错误模式 | 叙事闭合、过度自信 | 规则过重、摩擦大 |

所以 MASL 的存在理由不是“LLM 不会调度”。LLM 会调度。

MASL 的存在理由只能是：

```text
某些交付边界不应该每次都靠 LLM 临场想起来，而应该外化为可审计、可重放、可验证的制度。
```

如果这个理由不成立，MASL 就没有必要。

## 九、当前 MASL 的阶段意义

回到当前项目，MASL 仍在 labs 阶段。它不是已经证明的通用 agent runtime，也不是要替换 Codex、Claude Code、opencode 或 Gemini CLI。

当前做的很多东西，不一定是最终产品形态。它们更像实验探针，用来发现 AI 交付到底需要哪些外部认知结构。

`runtime-progress.json` 的意义，不只是显示系统在干什么，而是观察哪些执行状态对人类有用，哪些状态能提高人机交流带宽。

phase5 review loop 的意义，不只是“评审不过就重跑”，而是观察 truth failure 如何进入下一轮 LLM 上下文，并改变它的注意力中心。

owner-facing summary 的意义，不只是生成漂亮中文摘要，而是观察人类需要什么 truth projection：已证明什么、未证明什么、为什么停住、用户该决定什么。

phase-boundary safe-stop 的意义，不只是处理父进程旧模块，而是观察控制面自修改时，什么时候必须 fresh grounding，什么时候旧上下文不能继续被信任。

general intake 的意义，不只是让自然语言需求进入 live Codex，而是观察 raw prompt 如何变成可执行 truth hypothesis，哪些需求必须结构化停住，哪些可以低风险执行。

所以当前工作不是为了证明某个机制就是护城河，而是为了不断逼近这些问题：

```text
LLM 在真实交付中哪里会漂？
哪些 truth 需要外化？
哪些错误会重复出现？
哪些用户价值判断必须沉淀？
哪些观察机制能提高人机带宽？
哪些流程 runtime 很快就会吃掉？
```

如果未来 runtime 吃掉某层，MASL 不应该硬撑。它应该退到更合适的位置：project policy、schema、evaluation cases、failure taxonomy、用户价值判断库、工作流适配层。

## 十、下一刀的意义：truth failure -> context recompile

当前最值得做的一刀，仍然是：

```text
把上一轮 phase5 verdict / reason_codes / evidence_gaps 注入 auto-feedback loop 的 rerun phase prompts。
```

但它不能被解释成“强化 auto-feedback loop”。这还是执行层叙事。

更准确的解释是：

```text
验证一个 truth correction loop：
phase5 发现 truth failure；
系统能把 failure 编译成下一轮 LLM 的中心上下文；
并要求新的 evidence 来关闭 failure。
```

弱做法是：

```text
review failed -> append metadata -> rerun phase4
```

强做法是：

```text
review failed because evidence_gap X
X maps to implementation verification
therefore rerun phase4 with X as central objective
phase4 must produce evidence Y
phase5 must re-check X
old passing tests cannot close X unless they actually prove Y
```

这才是 MASL 和普通 `/goal` 的差异。

Runtime `/goal` 可以说：

```text
继续完成目标。
```

MASL 应该证明：

```text
根据上一轮 truth failure，重新编译下一轮 LLM 的目标、注意力、完成条件和裁决标准。
```

这个实验如果成立，说明 MASL 的 feedback compiler 有真实价值。如果不成立，说明我们只是在做字段透传。

## 十一、以后判断 MASL 功能的标准

以后每次想做 MASL 功能，都应该问：

```text
这件事是在增强 agent 执行能力，还是在增强 truth recovery / correction / projection？
```

如果只是增强执行能力，大概率会被 runtime 吃。

危险方向包括：

```text
更好地派子 agent
更好地拆任务
更好地 monitor progress
更好地续跑
更好地做 dashboard
```

这些不是不能做，而是要非常小心。除非它们直接服务 truth recovery、truth correction 或 truth projection，否则容易变成 runtime 未来原生能力的外部复制。

更值得做的是：

```text
从 raw prompt 提取验收口径
识别哪些需求必须追问
判断证据是否足够证明验收
把 review gap 精准映射回 target phase
把 owner action 变成状态迁移
把未证明事项明确展示给用户
积累 runtime 常见误判样本
沉淀用户价值判断和错误模式
```

也可以写成一张执行前审视卡：

```text
1. 这个功能沉淀了什么用户价值判断？
2. 它沉淀了什么错误模式？
3. 下次任务如何检索并注入这条沉淀？
4. 它有没有降低单次 LLM 推理负担？
5. 它有没有提高人机交流带宽？
6. 它有没有防止 narrative closure without evidence closure？
7. 如果 runtime 原生支持这个能力，MASL 应该退到哪里？
```

这张卡比“我们在做 phase-first 多 agent 系统”更能约束方向。

## 十二、可证伪信念

这篇文章不应该成为信仰宣言。它背后的信念必须能被证伪。

### 信念一：真实交付需要比聊天更强的可审计性

如果用户普遍接受 runtime 最后一段 summary 作为交付真相，不需要 evidence、review gap、owner action 和状态记录，那么 MASL 的 evidence ledger 会显得过重。

### 信念二：LLM 临场自组织在长链路里仍会不稳定

如果 runtime supervisor 能稳定从 raw prompt 到 owner-facing delivery，不忘验收、不误判证据、不丢 review feedback、不越过 owner boundary，那么外部 truth layer 的价值下降。

### 信念三：用户价值判断需要外部沉淀

如果 runtime memory 能稳定继承项目偏好、风险取舍和历史错误模式，而且不会过期污染，那么 MASL 的 memory/policy 层会被吃掉。

### 信念四：review failure 必须显式进入下一轮认知中心

如果 runtime 原生 review loop 能稳定把 evidence gap 转成下一轮执行中心，并逐项验证关闭，那么 MASL 的 feedback compiler 会被吃掉。

### 信念五：owner-facing truth 不是 transcript summary

如果用户接受“AI 总结一下”作为足够交付界面，不需要“已证明/未证明/为什么停住/下一步决策”的结构化展示，那么 owner-facing truth projection 的价值下降。

### 信念六：复杂任务通过细粒度拆解、持久化、重试可以降低单次 LLM 智力要求

如果更强模型直接端到端完成复杂交付，外部拆解主要增加摩擦，那么 MASL 的 phase-first 实验需要收缩。

这些信念如果被证伪，MASL 应该收缩，而不是继续扩大。

## 结语：从一念到交付之间

用户的一句话，是一念。  
真实交付，是现实世界里的结果。

一念到交付之间，有很长的距离。

这段距离里，有意图恢复，有假设显式化，有任务拆解，有证据收集，有 review failure，有反馈校正，有持久化，有重试，有人类价值判断，有 owner 决策，也有 LLM 很容易生成的漂亮叙事。

MASL 不应该把自己理解成更聪明的 agent。LLM 才是智能核心。runtime 会越来越会调度、越来越会记忆、越来越会自我修正。MASL 如果只是调 agent、拆 phase、写 prompt，它会被吃掉。

MASL 还能探索的，是另一件事：

```text
如何把低带宽 prompt 逐步外化成可检验假设；
如何把失败外化成修正信号；
如何把用户价值判断外化成稳定结构；
如何把复杂任务拆成足够小、可恢复、可验证的认知单元；
如何让人类看到 truth，而不是只看到叙事。
```

这不是对抗 LLM，也不是对抗 runtime。

这是在承认智能核心属于 LLM 之后，继续追问：智能如何穿过时间、错误、上下文限制和人类价值判断，抵达一个可证的交付结果。

如果未来 runtime 原生完成了这整条链路，MASL 应该退到 policy、schema、eval、knowledge asset。  
如果 runtime 还不能稳定做到，MASL 就仍然值得作为实验场，继续把那些还不能靠临场智能稳定完成的边界外化、沉淀、验证。

这就是当前阶段的意义。
