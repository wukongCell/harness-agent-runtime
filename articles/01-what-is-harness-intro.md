# 什么是 Harness：AI Agent 从会生成到能工作的中间层

最近讨论 AI Agent 和 AI coding，很容易出现一个问题：大家其实在讲不同的层。

有人在讲 prompt，关心怎么把指令写清楚。

有人在讲 context，关心怎么把项目背景、代码库、文档和历史信息塞给模型。

有人在讲 tools，关心模型能不能调用 shell、浏览器、数据库、IDE、MCP。

有人在讲 multi-agent，关心要不要把任务拆给多个角色。

还有人在讲 Codex、Claude Code、OpenClaw、Cursor、Goose、各种 skills 和 agent framework。

这些都重要，但它们不是同一件事。

如果要找一个更底层的问题，应该先问：

**模型已经会生成了，那它怎样才能真的工作？**

这里就需要一个词：**Harness**。

### 1. Harness 是什么

我现在对 Harness 的定义是：

**Harness 是把模型能力约束、连接、验证、回写成可重复工作系统的运行层。**

更口语一点说：

Prompt 让模型知道你想要什么。

Context 让模型知道现在发生了什么。

Tools 让模型能做事。

Harness 让模型做事时不乱来、能被验证、能被纠偏、能越用越好。

所以 Harness 不是一个更长的 prompt，也不是多几个 agent，更不是把工具一股脑塞给模型。它是一套运行环境和控制结构：任务怎么定义，事实从哪里来，工具能用到什么程度，权限怎么给，输出怎么验收，错误怎么恢复，人的反馈怎么沉淀到下一次。

一个真正有用的 agent 系统，不只是“模型 + 工具”。更准确地说，它应该是：

**模型 + 上下文 + 工具 + 权限 + 工作流 + 验证器 + 状态 + 回写机制。**

这整套东西合起来，才是 Harness。

### 2. 为什么现在才开始需要这个词

早期用大模型，问题主要是“怎么让它答得更好”。

所以大家先研究 prompt engineering：怎么写角色，怎么拆步骤，怎么给例子，怎么让模型输出固定格式。

后来模型开始进入更复杂任务，问题变成“怎么让它知道更多背景”。

所以大家开始讲 context engineering：怎么组织文档，怎么检索代码，怎么压缩历史，怎么把当前环境喂给模型。

但 AI Agent 和 AI coding 把问题又往前推了一层。

现在模型不只是回答问题，而是在真实环境里做事：改代码、跑测试、点浏览器、查日志、开 PR、写文档、重构工程、处理长任务。

这时真正的瓶颈不再只是“它知不知道”，而是：

- 它做错了谁来发现？
- 它怎么知道自己已经完成？
- 它能不能访问正确的事实来源？
- 它有没有权限边界？
- 它跑偏后怎么停下来？
- 人的修改能不能变成下一次的规则？
- 它能不能跨几个小时持续推进，而不是中途漂移？

这些问题不是 prompt 能单独解决的，也不是换一个更强模型就会自动消失的。

因为这已经不是生成问题，而是运行问题。

Harness 解决的就是运行问题。

### 3. AI coding 为什么最先暴露 Harness 问题

AI coding 是最适合理解 Harness 的场景，因为代码世界天然有一套真实环境。

一个 coding agent 面对的不是空白文本框，而是一个 repo。

repo 里有源代码、README、架构约束、测试、lint、类型检查、构建脚本、浏览器页面、CI、git diff、PR review、部署环境。

这些东西构成了一个非常好的 Harness 基础。

如果没有这个运行层，模型再会写代码，也只是“生成一段看起来像代码的文本”。它可能一次写对，也可能写出能骗过自己但跑不起来的东西。

但一旦把模型放进 Harness，事情就变了：

- 它不能只说“我觉得对”，要跑测试
- 它不能只说“页面应该能用”，要用浏览器点一遍
- 它不能只凭记忆猜项目结构，要读 repo 里的事实来源
- 它不能随便改所有文件，要遵守架构边界和权限
- 它不能每次重复犯同一个错，人的 review 要回写成规则、文档或 skill

这也是为什么 AI coding 工具的发展，很快就从“代码补全”走向“agent 工作环境”。

真正的分水岭不是模型能不能写代码，而是它有没有被放进一套能约束、验证和纠偏的工作系统。

### 4. Harness 不是多 agent

很多人一听 agent workflow，就会自然想到 multi-agent。

这很容易误导。

多个 agent 只是 Harness 可能使用的一种结构，不是 Harness 本身。

如果没有外部反馈，多 agent 可能只是把同一种错误并行放大。

如果没有状态管理，多 agent 会互相覆盖、重复劳动、争抢上下文。

如果没有清晰分工，一个 agent 同时当 planner、coder、reviewer、operator，也会出现职责过载；多个 agent 如果角色边界不清，只是把这个问题放大。

所以判断一个 agent 系统质量时，不该先问“有几个 agent”，而该先问：

- 外部反馈在哪里？
- 验收标准在哪里？
- 状态如何交接？
- 权限边界在哪里？
- 错误如何被记录？
- 有效经验如何回写？

这些问题回答不清楚，再多 agent 也只是更复杂的 demo。

Harness 的重点不是“任务编排”，而是“反馈编排”。

### 5. Harness 的核心部件

一个成熟的 Harness，至少包含几类东西。

第一类是**事实来源**。

Agent 必须知道什么是权威信息。对 coding agent 来说，可能是 repo、AGENTS.md、CLAUDE.md、README、架构文档、issue、测试结果。对研究 agent 来说，可能是原始论文、资料库、引用链和保存下来的 raw evidence。

如果事实不在系统里，agent 就无法稳定使用。对 agent 来说，看不到的东西基本等于不存在。

第二类是**任务边界**。

任务不能只是“帮我优化一下”。好的 Harness 会把目标、范围、非目标、验收标准、风险边界讲清楚。尤其是长任务和复杂任务，边界越模糊，agent 越容易在看似努力中跑偏。

第三类是**工具和权限**。

Agent 能调用什么工具，能不能写文件，能不能联网，能不能执行命令，能不能改数据库，能不能部署线上环境，这些都不是小细节。工具给得太少，agent 做不了事；工具给得太宽，系统风险会急剧上升。

第四类是**外部验证器**。

这是 Harness 最关键的部分之一。

模型自评在结构上是不可靠的。生成者天然容易宽容自己的产出。尤其是复杂任务里，它经常能给出一套听起来合理的解释，但真实环境不买账。

所以高质量 Harness 一定会把验证尽量搬到模型外部：

- 测试是否通过
- 类型检查是否通过
- lint 是否通过
- 页面能不能真实点击
- API 是否返回正确结果
- CI 是否通过
- 人类 review 改了哪里

外部验证不是为了让系统显得严谨，而是为了让 agent 接触真实反馈。

第五类是**状态和记忆**。

Agent 做长任务时，最怕每一轮都像重新开始。它需要知道当前任务推进到哪里，哪些尝试失败过，哪些决策已经做过，哪些约束不能再碰。

但记忆不能只是无限堆聊天记录。好的 Harness 会区分临时上下文、任务状态、长期规则、项目事实和可复用经验。

第六类是**回写机制**。

这是很多系统真正缺失的地方。

如果一次 review 只改了这次输出，但没有把原因写回规则、文档、测试、模板或 skill，那么下一次 agent 还会犯同样的错。

反馈只有回写，才会变成系统能力。

否则系统每次都像失忆一样重新开始。

### 6. Skills 为什么属于 Harness 问题

现在很多人把 skills 理解成高级 prompt 库，这个理解太浅了。

真正有价值的 skill，不只是几段提示词，而是一个能力包。

它可能包含说明文档、脚本、模板、检查清单、示例、数据、触发条件、注意事项，甚至一套完整工作流。

换句话说，skill 是把一个可重复的工作方式产品化。

这也是为什么 skills 和 Harness 的关系很近。

Harness 负责提供运行层，skills 则把运行层上的高价值操作打包成可复用能力。

一个好的 skill 会告诉 agent：

- 什么时候该用它
- 应该读哪些材料
- 应该按什么步骤做
- 可以调用哪些脚本
- 输出应该长什么样
- 常见错误是什么
- 最后如何验证

这已经不是 prompt engineering，而是 workflow engineering。

当一个团队把 review 习惯、测试方法、发布流程、设计判断、写作规范都沉淀成 skills，它其实是在把人的工作方法变成 Harness 的一部分。

### 7. Harness 不是越厚越好

讲到这里，很容易走向另一个误区：既然 Harness 很重要，那是不是越复杂越好？

不是。

Harness 的厚度要和任务复杂度匹配。

简单任务不需要重型 Harness。让模型改一个小文案、解释一段代码、生成一个临时脚本，薄一点的约束就够了。

复杂任务才需要更强的 Harness。比如跨文件重构、长时间自动开发、多角色协作、生产环境操作、金融决策、研究工作流、持续维护知识库。这类任务里，真正危险的不是模型不会生成，而是它在长链路中缺少反馈、状态、权限和验收。

所以“Big Model vs Big Harness”不是一个真正的二选一问题。

更好的问法是：

**这个任务是否已经超过单次生成的可靠边界？**

如果没有，薄 Harness 足够。

如果超过了，就必须把验证、约束、状态和回写认真设计出来。

### 8. 对 agent 产品意味着什么

理解 Harness 后，再看 agent 产品，判断标准会变得很不一样。

很多 agent 产品看起来是在卖“更聪明的助手”，但真正的壁垒可能不是模型调用，而是默认 Harness 设计。

它有没有理解用户的真实工作环境？

有没有把任务边界设计清楚？

有没有接入正确工具？

有没有可观测状态？

有没有可靠验证？

有没有失败恢复？

有没有把用户反馈转成长期能力？

有没有把高频工作流打包成可复用资产？

这些问题，比“用了哪个模型”更接近产品本质。

因为模型会持续变强，工具接口会持续变化，但一个领域里的工作流、约束、验收方式和组织经验，才是更长期的复利来源。

未来很多 agent 产品的差异，可能不在“谁能调用模型”，而在“谁更懂某类任务应该被怎样托管、验证和改进”。

### 9. 对个人意味着什么

对个人使用者来说，Harness 的启发也很直接：

不要只问“我该怎么提示 AI”。

要开始问：

- 我的工作事实来源在哪里？
- 我的项目规则有没有写下来？
- 哪些任务可以被稳定交给 agent？
- 我怎么验收 agent 的结果？
- 我每次修改 agent 输出的原因，有没有沉淀成规则？
- 哪些重复流程可以变成 skill？
- 哪些知识可以变成 agent 可用的 context infrastructure？

你会发现，真正提高 AI 使用水平的，不是收藏更多 prompt，而是把自己的工作环境整理成 agent 能读、能用、能验证、能回写的系统。

这也是 AI coding 给普通人的最大启发：

以后重要的能力，不只是“会不会和 AI 聊天”，而是“会不会搭一套让 AI 真能工作的环境”。

### 结语

如果只用一句话总结：

**Harness 是 AI Agent 从“会生成”走向“能工作”的中间层。**

它把模型放进真实世界，让它面对事实、边界、工具、权限、反馈、验证和状态。

没有 Harness，agent 很容易停留在看起来很聪明的生成器。

有了 Harness，agent 才开始接近一个能被托管、能被纠偏、能被复用、能持续改进的工作系统。

所以未来讨论 AI Agent，不该只问模型有多强，也不该只问有几个 agent。

更关键的问题是：

**你给它建了一个怎样的工作环境？**
